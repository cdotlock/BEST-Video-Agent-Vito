# PRD：BEST Video Agent 从单一小说视频工作流升级为通用视频平台

- 版本：v0.1（待评审）
- 日期：2026-03-04
- 状态：Draft
- 范围：仅产品与架构方案，不含代码改动

## 1. 背景与问题

当前系统已经具备「Agent + MCP + Skills + 资源面板 + 任务流」的完整基础设施，但在产品形态上仍是“小说转视频”专项工作台，存在以下核心问题：

1. 工作流入口和上下文强绑定 `novelId/scriptKey`，对通用视频任务（广告、短视频、IP 混剪、品牌素材）不友好。
2. 画风控制主要分散在 prompt/skill 文本中，缺少可复用、可追踪、可迭代的“风格资产层”。
3. 生成前初始化不足，用户难以在开始前确认视觉方向，容易反复返工。
4. UI 当前是深色、工程导向的信息布局，视觉高级感与操作效率都不足。
5. 技能与能力暴露过多，普通用户的主路径被高级功能干扰。
6. 当前“记忆”主要是会话上下文与 tool recall，不是“用户偏好可积累的长期记忆”。

## 2. 产品目标

## 2.1 目标（Must Have）

1. 将“单一小说视频平台”升级为“通用视频平台”，支持多类型视频创作场景。
2. 在生成前提供“风格初始化向导”：搜图、挑图、参考图分析、提示词反推、风格确认。
3. 完成一次大规模 UI/UX 升级：白色主基调、简洁高效、明显二级菜单、直观素材展示。
4. 缩减大多数用户用不到的 Skills/MCP 暴露，保留核心路径。
5. 建立“可记忆、可进化”的闭环，让系统越用越贴近用户画风与流程偏好。
6. 增加执行模式开关：`Checkpoint`（关键步骤确认）与 `YOLO`（自动推进、无需确认）。

## 2.2 非目标（当前阶段不做）

1. 不在本阶段重做底层 MCP 协议或替换现有任务流架构。
2. 不在本阶段引入多租户权限系统（先沿用当前 user name 维度隔离）。
3. 不在本阶段引入复杂审批流或多人协同工作流。

## 3. 现状诊断（基于当前代码）

1. 专项绑定明显：`src/app/video/*` 与 `video_context` 以 `novelId/scriptKey` 驱动。
2. 前端主色为深色（`bg-slate-950` + Antd darkAlgorithm），与目标白色简洁风冲突。
3. 视频页默认预加载 `novel-video-workflow`、`novel-character-card`，存在领域耦合。
4. `video_mgr__generate_image` 已支持 `referenceImageUrls`，但缺少“搜索参考图 + 风格归纳 + profile 固化”前置链路。
5. KeyResource 有版本化（prompt/回滚/再生），但缺少跨 session 的偏好学习与策略沉淀。
6. 主站资源抽屉暴露全部 skills/mcps，对普通用户心智负担较大。

## 4. 目标用户与核心场景

1. 普通创作者：想快速产出风格稳定的视频素材，不关心底层技能系统。
2. 内容团队：希望团队内复用统一风格模板，减少 prompt 试错。
3. 高级用户：仍可进入高级模式，使用 MCP/Skill 做深度定制。

核心场景：

1. 新项目创建 -> 风格初始化 -> 分镜素材生成 -> 视频提示词/视频生成 -> 成果沉淀。
2. 老项目复用 -> 直接套用历史风格档案 -> 少量微调 -> 快速批量产出。

## 5. 目标方案总览

## 5.1 产品结构（通用工作台）

将现有“小说专项页”升级为“通用项目工作台”，核心对象从 `novel -> episode` 抽象为：

1. `Project`：项目容器（任务类型、交付目标、默认风格）。
2. `Sequence`：序列单元（可对应章节/镜头组/场景段/广告片段）。
3. `DomainResource`：素材统一表（沿用 `domain_resources`，按 `category + media_type` 渲染）。
4. `StyleProfile`：风格配置资产（新增）。
5. `PreferenceMemory`：用户长期偏好记忆（新增）。

## 5.2 生成前初始化（风格向导）

新增“开始生成前必须走的初始化流程”，默认 4 步：

1. 明确目标：视频用途、受众、时长、平台、比例。
2. 搜索参考图：基于关键词和语义扩展，从公共图库（Pexels/Unsplash 等）检索可用图片。
3. 风格反推：对选中参考图做视觉分析，提取 style tokens 和可用 prompt。
4. 风格确认：生成 `StyleProfile`（正向 prompt、负向 prompt、色彩/镜头语言/材质倾向）。

执行模式规则：

1. `Checkpoint` 模式：初始化结束后及关键生成节点需要用户确认再继续。
2. `YOLO` 模式：初始化完成后自动推进后续流程，不等待确认。

## 5.3 提示词系统通用化

把现有“隐式风格 prompt”升级为“分层 prompt 组装”：

1. Base Layer：质量与安全基础约束。
2. Style Layer：来自 `StyleProfile` 的稳定风格约束。
3. Content Layer：当前序列剧情/镜头内容。
4. Task Layer：图像或视频生成任务特定参数。

效果：任何画风都通过 Style Layer 注入，不再靠硬编码“某一种风格”。

## 6. 信息架构与 UI/UX 升级方案

## 6.1 视觉方向

1. 主色调：白色与浅灰为主，高信息区域用低对比分层。
2. 风格关键词：简洁、专业、轻量、可控。
3. 保留重点色：仅用于状态和关键操作（如生成中、失败、已确认）。

## 6.2 页面布局（桌面优先，移动自适应）

1. 顶部：项目级导航（项目切换、风格库、导出、设置）。
2. 左侧一级菜单：项目、序列、风格库、历史成果。
3. 左侧二级菜单：在当前一级菜单下展开过滤器与快捷操作。
4. 中央工作区：对话与任务执行主区（支持“聊天 / 时间线 / 分镜”切换）。
5. 右侧素材区：按 `category` 动态分组展示图片、视频、JSON，并支持筛选、排序、预览、版本操作。

## 6.3 核心交互升级

1. 新建项目默认进入“风格初始化向导”。
2. 素材卡片支持一键：设为风格参考、复制 prompt、再生成、对比版本。
3. 所有高频操作显式按钮化，低频操作收纳到二级菜单或更多面板。
4. 支持快捷键（发送、撤销、切换视图、聚焦搜索）。
5. 移动端简化为“单列分步工作流”，侧栏改为抽屉。
6. 顶部提供 `Checkpoint / YOLO` 模式切换，并在状态栏显示当前模式。

## 7. Skills/MCP 精简策略

## 7.1 直接精简策略

默认仅保留：

1. 多模态生成相关（video_mgr + upload + 必需上下文能力）。
2. 项目与风格管理相关工具。

默认下线（不再面向普通产品路径暴露）：

1. dynamic-mcp-builder
2. business-database
3. api-builder
4. langfuse-admin 等非核心用户路径能力

说明：不是“先隐藏”，而是直接从默认产品能力集中精简移除，仅保留运营/开发内部入口。

## 8. 记忆与自进化方案

## 8.1 记忆分层

1. 会话记忆：当前已存在（ChatSession + ChatMessage + recall）。
2. 项目记忆：项目级风格和流程偏好。
3. 用户长期记忆：跨项目风格偏好、常用镜头语言、常用参数。
4. 结果记忆：被用户采纳的资源及其 prompt 特征。

默认策略：

1. 长期记忆默认开启（用户首次使用即生效）。
2. 提供显式开关用于手动关闭，但默认值保持开启。

## 8.2 记忆写入触发点

1. 用户选择参考图并确认风格（Checkpoint）或自动接受风格（YOLO）时。
2. 用户保存 prompt、再生成、回滚版本时。
3. 用户标记“满意/不满意”或导出最终素材时。

## 8.3 自进化机制

1. 事件采集：记录关键行为（选择、修改、采纳、否定）。
2. 偏好提炼：将事件归纳为结构化偏好 token（带置信度和时间衰减）。
3. 下次启动：自动推荐“猜你想要”的风格模板与流程默认值。
4. 用户可控：提供“查看记忆、删除记忆、重置偏好”能力。

## 9. 数据模型建议（非破坏性增量）

原则：尽量复用现有 `domain_resources`，新增表承载“项目、风格、记忆”。

建议新增（命名可调整）：

1. `VideoProject`：项目元信息与默认配置。
2. `VideoSequence`：序列单元，替代硬编码 `novel_scripts` 语义。
3. `StyleProfile`：风格档案（prompt 核心字段 + style JSON）。
4. `StyleReference`：风格参考图来源、版权与分析结果。
5. `UserPreferenceMemory`：用户偏好 token 与置信度。
6. `GenerationFeedback`：采纳/再生/回滚/评分事件。
7. `ExecutionModeConfig`：项目级执行模式配置（`checkpoint` / `yolo`）。

备注：全部为新增表与新增字段，避免破坏性迁移。

## 10. API 与 MCP 规划

## 10.1 API（建议）

1. `POST /api/video/projects`：创建通用项目。
2. `GET /api/video/projects`：项目列表。
3. `POST /api/video/projects/{id}/sequences`：创建序列。
4. `POST /api/video/style/search`：搜图入口。
5. `POST /api/video/style/reverse`：参考图反推风格。
6. `POST /api/video/style/profiles`：保存风格档案。
7. `GET /api/video/style/profiles`：读取可复用风格档案。
8. `POST /api/video/memory/feedback`：写入偏好反馈事件。
9. `PATCH /api/video/projects/{id}/execution-mode`：切换 `checkpoint` / `yolo`。

## 10.2 MCP（建议）

1. `style_search__search_images`
2. `style_search__analyze_references`
3. `style_profile__create`
4. `style_profile__list`
5. `style_profile__apply`
6. `video_memory__record_feedback`
7. `video_memory__recommend_defaults`

所有入口遵循当前约定：API/MCP 仅做协议层 + Zod 校验，业务逻辑统一进 service layer。

## 11. 分阶段实施计划

## Phase 0：架构准备（不改变用户路径）

1. 引入 `Project/Sequence/StyleProfile/Memory` 的 service 与 schema（增量）。
2. 引入执行模式配置（`checkpoint` / `yolo`）。
3. 完成基础 API 与最小 MCP 接口。

验收：现有流程不回归，新数据模型可写可读。

## Phase 1：风格初始化链路

1. 接入公共图库搜索能力（含来源信息与 license 元数据）。
2. 完成参考图反推与 style profile 生成。
3. 将 profile 注入生成链路，替换固定风格 prompt。
4. 接入 `YOLO / Checkpoint` 分支执行逻辑。

验收：用户可完成“搜图 -> 选图 -> 生成”，且 `YOLO` 模式下不出现确认阻塞。

## Phase 2：白色高效 UI 改版

1. 重做布局与设计 token。
2. 上线二级菜单与素材可视化升级。
3. 优化核心高频交互与移动端适配。

验收：主流程操作步数下降，核心操作可达性提升。

## Phase 3：能力收敛与分级暴露

1. 下线低频 Skills/MCP 的产品入口。
2. 精简系统 prompt，仅注入保留能力。
3. 保留内部运维入口用于高级调试。

验收：普通用户路径更短，能力面板仅保留必要项。

## Phase 4：记忆与自进化闭环

1. 上线反馈事件写入与偏好聚合。
2. 在初始化和生成阶段做个性化默认值推荐。
3. 提供记忆可视化与重置。

验收：老用户二次创作的手动调整量显著下降。

## 12. 指标与验收标准

产品指标：

1. 风格确认成功率：首次生成后无需大改的比例。
2. 首轮可用率：首轮产出被采纳为候选素材的比例。
3. 迭代效率：从创建项目到第一条可用视频提示词的平均时长。
4. 用户负担：平均会话中的手工 prompt 修改次数。
5. 记忆收益：老用户相比新用户的平均迭代轮次下降幅度。

工程指标：

1. API 与 MCP 入口全部走 service layer。
2. 输入全部 Zod 校验。
3. 保持 strict TS 与无 `any`。
4. 文档同步：`docs/api-playbook.md` + 对应 `docs/useCase/*`。

## 13. 风险与应对

1. 搜图版权风险：强制保留来源与 license 信息，仅收录可商用来源或标注风险。
2. 风格反推不稳定：采用“结构化风格 token + Checkpoint 显式确认 / YOLO 自动接受”双轨，不直接黑箱替换。
3. 记忆污染：偏好写入必须基于用户明确行为（采纳/修改/回滚），并提供可回滚。
4. UI 大改带来学习成本：保留“经典布局开关”作为短期过渡。
5. YOLO 误触发风险：在模式切换处给出显式提示并允许一键降回 Checkpoint。

## 14. 与现有约束的对齐

1. 保持 MCP 标准与 asMCP 能力，不引入私有协议。
2. 所有新增能力放在 service layer，API/MCP 仅做协议转换。
3. 继续使用 `domain_resources` 作为统一素材表，避免回退到多张业务碎表。
4. 保留 Context Recovery 原则：关键中间产物必须持久化，避免长对话丢失。

## 15. 评审待确认项

1. 公共图库首批接入范围（Pexels、Unsplash、Pixabay）的优先级与配额策略。
2. YOLO 模式下哪些节点仍需强制安全确认（如外部发布、批量覆盖）。
3. Skills 下线清单与“内部运维入口”的权限边界。
